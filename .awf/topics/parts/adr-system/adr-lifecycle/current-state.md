The adr package parses decision records, derives their identity, and validates their lifecycle. The claims below capture the current parsing and status contracts.

## Claims

### `invariant: fresh-adoption-v1-cutoff`

Empty first adoption seals both ADR format cutoffs at 1; brownfield first adoption seals both at the highest existing identity plus one with every lower gap explicit; upgrading an existing V1 adopter preserves its V1 cutoff and seals the V2 cutoff at the highest identity plus one; every ADR is legacy below V1, V1 from that cutoff to before V2, and V2 at or above the V2 cutoff.
Origin: ADR-0139
Revised-by: ADR-0143
Backing: test

### `invariant: adr-new-heading-matches-file`

A file created by awf new adr carries a # ADR-NNNN: <title> heading whose number matches the NNNN prefix of its own filename, and the literal Title placeholder never survives into the written file.
Origin: ADR-0042
Backing: test

### `invariant: adr-new-no-overwrite`

awf new adr refuses to overwrite an existing file at its computed target path, returning an already-exists error rather than clobbering it.
Origin: ADR-0042
Backing: unbacked
Verify: Read NewFile in internal/adr/adr.go and confirm it stats the computed target path and returns an already-exists error before performing any write.

### `invariant: adr-new-sequential-numbering`

NextNumber returns the highest existing ADR number in the decisions directory plus one, and never reuses a number already present in that directory.
Origin: ADR-0042
Backing: test

### `invariant: adr-new-strips-markers`

A file created by awf new adr never contains a GENERATED by awf or awf:edit marker comment line.
Origin: ADR-0042
Backing: test

### `invariant: adr-related-ascending`

Every ADR's related array must ascend, and awf check reports an adr-related-order finding naming the first descent when one does not; link resolution is checked independently of ordering, so a descending array still has every entry verified against the corpus.
Origin: ADR-0131
Backing: test

### `invariant: adr-related-link-resolved`

awf check fails an ADR whose related list names an ADR number with no matching file under the decisions directory, independent of any tag vocabulary.
Origin: ADR-0103
Backing: test

### `invariant: adr-sections-parsed`

Parsing an ADR directory splits each document body on its `## ` headings and populates a per-heading section map, so a caller can read the full text of a named section such as Invariants directly from the parsed record.
Origin: ADR-0007
Backing: test

### `invariant: adr-status-enum-and-matrix`

Every governed ADR is routed by the two immutable format cutoffs: V1 retains its four statuses and five legal edges, while V2 recognizes Proposed, Accepted, Implementing, Implemented, and Abandoned and accepts only the format-specific status, history-event, digest, and application-cardinality transitions.
Origin: ADR-0135
Revised-by: ADR-0143
Backing: test

### `invariant: applied-history-events-append-only`

Stable V2 Status history is prefix-append-only: each Applied event records one nonempty, declaration-ordered batch of previously unapplied operations with one positive state sequence, and a checked pair refuses deletion or mutation of any prior event.
Origin: ADR-0143
Backing: test

### `invariant: audit-shares-adr-parser`

internal/audit parses ADR frontmatter through internal/adr's exported bytes-level parsing seam and declares no ADR frontmatter struct of its own.
Origin: ADR-0130
Backing: test

### `invariant: corpus-owns-field-reads`

No file outside internal/adr reads ADR.Sections directly; parsed section questions are answered through internal/adr methods rather than re-derived by consumers.
Origin: ADR-0130
Revised-by: ADR-0138
Backing: test

### `invariant: corpus-owns-status-literals`

No non-test production file outside internal/adr compares an ADR status against a string literal or tests it with a status prefix, except awf context's enumerated Tier-2 exclusion.
Origin: ADR-0130
Backing: test

### `invariant: corpus-parsed-once`

adr.ParseDir has no production call site outside internal/adr, and inside internal/adr only the corpus construction seam and NextNumber call it; consumers do not independently parse the decisions directory.
Origin: ADR-0130
Revised-by: ADR-0138
Backing: test

### `invariant: corpus-raw-access-enumerated`

The two ordered schema migrations in internal/migrate are the only call sites of the corpus view's raw-bytes accessor, and no file outside internal/adr calls os.ReadFile on an ADR.Path.
Origin: ADR-0130
Revised-by: ADR-0138
Backing: test

### `invariant: corpus-single-identity-key`

Every parsed ADR's identity key is the four-digit number taken from its NNNN-*.md filename prefix, and a file whose name does not match that pattern is not parsed as an ADR.
Origin: ADR-0130
Backing: test

### `invariant: decision-index-is-historical-not-authoritative`

The generated decision index separates in-flight work from compact history and never renders supersession or currentness inference.
Origin: ADR-0135
Backing: unbacked
Verify: Fixtures rendered in every legal status produce the exact INDEX.md sections with no anchor annotations.

### `invariant: decision-items-enumerable`

Parsing an ADR fails when its Decision section has no column-0 numbered items or when its item numbers are not sequential from 1 (a gap, duplicate, or restart).
Origin: ADR-0120
Backing: test
