## Current state

ADRs are append-only history, not active authority: active rules and invariants live as current-state topic claims. Two immutable lock cutoffs route records: below `adrFormatV1From` is legacy, from V1 to before `adrFormatV2From` is the four-state V1 grammar, and at or above V2 is `current-state-v2`. Both governed formats use the ordered Context, Decision, State changes, Consequences, Alternatives Considered, and Status history sections. Decisions live under `docs/decisions/`, parsed by `internal/adr` and rendered into the generated drift-checked `INDEX.md`.

V2 statuses are Proposed, Accepted, Implementing, Implemented, and Abandoned. Its history is a date-nondecreasing prefix-append-only event stream. Status events carry the frozen content digest. Applied events use the exact grammar `- <date>: Applied; state-sequence: <positive integer>; operations: <declaration-ordered operation-list>`. Entering Implementing appends status then a first nonempty strict-subset batch, a middle pair appends one batch, and the final pair appends the remainder then Implemented. Direct terminal transitions use one implicit batch. Abandonment preserves applied operations, cancels the remainder, and carries a rationale. Stable history is corrected forward through a successor ADR.

The `## State changes` section is the authoritative relationship between an ADR and the topics it governs: either `None.` or a list of `- add`, `- update`, and `- remove` entries, each naming one claim by its qualified `<domain>/<topic>:<slug>` id. awf derives the affected topics and domains from these operations, so a v1 ADR carries no `topics`, `related`, `tags`, or `domains` frontmatter; a rename is a remove plus an add, a split is one remove plus several adds, and a removed id is never reused. For every newly appended implicit or explicit application batch, `awf check --staged` compares HEAD with the index and verifies exactly that transaction: an `add` claim appears with the ADR as `Origin`, an `update` preserves the claim's `Origin` and prior `Revised-by` prefix, appends the ADR once, and changes a canonical field, and a `remove` leaves no active claim. Static `awf check` validates the lifecycle, sequence, operation history, claim provenance, digest, and absence rules; the first v1 update or remove for an identity sealed into the migration baseline is allowed to begin without a v1 add operation, while every claim introduced after the cutoff still begins with exactly one add. `awf audit` and implementation review evaluate every parent-to-commit snapshot pair in an explicit range rather than only its endpoints.

Append-only protects rationale, not orthography (ADR-0115, widened by ADR-0118): an ADR's arguments, decision items, alternatives, and consequences are never rewritten, but a meaning-preserving punctuation normalization of its authored body is permitted, proven word-preserving by an alphanumeric-run comparison that must report zero delta. A live body's meaning is otherwise frozen, though a meaning-preserving schema retrofit may migrate its machine-readable encoding, the license under which the project-atomic upgrade rewrote the legacy corpus into the v1 format.

`INDEX.md` always renders, with placeholder sections when no ADR is in flight or recorded, so its document-map link resolves out of the box (ADR-0020). It lists Proposed, Accepted, and Implementing ADRs under an In flight section and Implemented and Abandoned ADRs under a compact History by number, title, and status, and renders no supersession chains, anchor annotations, currentness inference, or per-domain ADR index; generated domain docs link a compact topic list instead. A new ADR is scaffolded via `awf new adr "<title>"` (ADR-0042), which computes the next sequential number and copies `template.md` with its marker comments stripped and its date and title filled in; `awf-proposing-adr` invokes it as the first authoring step. The ADR guide (`README.md`), the skeleton (`template.md`), and a plan-authoring guide (`plans/README.md`) are awf-managed singletons (ADR-0021, ADR-0020), so the `.layout.adrReadme` / `.layout.adrTemplate` / `.layout.plansReadme` references resolve in any adopter project. The workflow chain (brainstorm → ADR → plan → resync → impl → review) is enforced by the `awf-*` skills and reviewer agents.
