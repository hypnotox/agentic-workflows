## Current state

Per-project configuration lives in a `.awf/` tree (relocated out of the runtime's `.claude/` directory by ADR-0016, schema migration `{To:3}`): a skeleton `config.yaml` (prefix, vars, flat enable arrays for skills/agents/docs/domains, the adapter `targets` array, and invariants) plus per-artifact sidecars and convention parts. The config is strict-parsed (`KnownFields`), and drift is tracked in a schema-versioned `awf.lock`. Schema migrations are an ordered registry applied by `awf upgrade`; `awf sync`/`check` gate a stale layout (a pre-relocation `.claude/awf/` tree is detected and gated until upgraded). Schema migration `{To:4}` dropped the `hooks:` enable array with the hook kind (ADR-0032). Schema migration `{To:5}` added the `bootstrap` enable entry ŌĆö a nested `bootstrap.enabled` scalar rather than a top-level array ŌĆö with the self-pinning bootstrap (ADR-0040). ADR-0048 reintroduced the `hooks:` key as a nested `hooks.enabled` mapping for the rendered git-hook payloads ŌĆö parse-safe because the schema-4 migration strips only the legacy array shape ŌĆö with no accompanying migration: `awf init` scaffolds it enabled, existing adopters opt in via `awf add hooks`. Additive optional fields (like `domains`) are backward-safe and need no version bump. The `config.yaml` that `awf init` scaffolds enables a curated workflow-core set (ADR-0022) ŌĆö only the catalog's `core`-flagged skills, plus all agents ŌĆö while seeding every template-referenced var (across all template families, including the always-on plain singletons) so a later opt-in `awf add` renders cleanly. No doc carries `core` any longer: the three docs that used to (`workflow`, `doc-standard`, `agents-md-standard`) are mandatory singletons outside the toggleable `docs:` catalog (ADR-0043), and a schema migration (`{To: 6}`) relocates their sidecar/convention-part paths and strips them from an upgrading project's `docs:` array. Those fillable values are declared as descriptors in the catalog's `vars:` block (ADR-0029) ŌĆö `{key, kind, description, default, options, target}` ŌĆö and `ScaffoldConfig` takes the resolved var values plus an optional invariants config, so a populated `config.yaml` (including the `invariants` marker/globs) can be written at init rather than only blank vars. The `vars:` block also carries two `multiselect` catalog-trim descriptors (`skills`/`docs`, `target: catalog-skills`/`catalog-docs`) whose options derive from the catalog; `ScaffoldConfig` takes an optional `*config.CatalogTrim` that replaces the core skills/docs enable arrays verbatim when supplied (nil keeps the curated core). The four catalog/domain enable arrays are managed through `awf add`/`remove <kind> <name>` and surfaced by `awf list` (ADR-0024); the editor is block-scoped, so a name shared across kinds (e.g. a `debugging` skill and doc) is toggled in the right array only. The adapter `targets` array has a parallel bespoke `awf add/remove/list target` path (ADR-0037) ŌĆö it is not a catalog kind, so it stays out of the kind-descriptor machinery and refuses removing the last target. `config.yaml` is constructed and mutated only through `internal/config` (ADR-0026): `MarshalSkeleton` emits the scaffold, `SetArrayMember` edits one array member, `SetArray` (ADR-0037) writes a whole computed list, and `SetMappingScalar` (ADR-0040) writes a nested `bootstrap.enabled` bool ŌĆö so adding `cursor` to an absent, defaulted `targets` key materializes `[claude, cursor]` rather than dropping the default ŌĆö all behind a single two-space `encode` funnel that retires the former hand-rolled emitter and string editor (`internal/migrate`'s forward-compat marshalling excepted). The `skills`/`agents` enable arrays may also carry project-local names absent from the catalog (ADR-0068): such a name must be declared by a `.awf/<kind>/<name>.yaml` sidecar (its `data.description` feeding the rendered frontmatter) and pass `config.ValidateArtifactName`, which enforces kebab-case so a name cannot smuggle path or template metacharacters; `awf new skill|agent` scaffolds exactly that declaration. Sidecar `data:` keys are now per-key overrides over catalog defaults ŌĆö an absent key falls through, a present key (even null or empty) suppresses the default (ADR-0045) ŌĆö and the skills enable array feeds the render context and the config hash of `.skills`-referencing templates, so toggling a skill reflags dependent artifacts (ADR-0046); no schema bump, the tree's shape is unchanged. Reviewing skills are pair-validated with the reviewer agents they dispatch (ADR-0050): a catalog `requiresAgent` on each `reviewing-*` spec fails every gated command at project open when an enabled, non-local reviewing skill's agent is missing from the `agents:` array, `awf remove agent` refuses upfront while an enabled skill requires the agent, and `awf add skill` enables a missing required agent in the same config rewrite. Commit scopes have one storage (ADR-0051): the `commitScope` var is gone from the catalog; a `commitScopes` init descriptor (`target: audit-scopes`) routes a comma-separated answer into `audit.allowedScopes`, which `awf commit-gate`/`awf audit` and the rendered reviewing-skill prose both consume through `audit.Resolve`, and `awf init` writes the block only for a non-empty answer (nil = accept any). The lock and every migration rewrite of an existing config file are written atomically ŌĆö a same-directory temp file renamed into place (`manifest.WriteFileAtomic`, ADR-0076) ŌĆö so an interrupted process can no longer truncate a trust-bearing file; and a present-but-unreadable `awf.lock` is a hard error in every reader via the `manifest.LoadOptional` choke point (one recovery hint: restore from version control or delete deliberately to re-adopt), with a missing lock keeping each caller's no-lock semantics.
