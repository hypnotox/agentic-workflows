## Current state

Each machine-enforceable ADR Invariant bullet is declared in one of two forms (ADR-0105): a backed `invariant: <slug>` (which requires a proof `<marker> invariant: <slug>` comment on a **test**) or an `unbacked-invariant: <slug>` carrying a `**Verify:**` note, a reasoned contract deliberately exempt from the proof requirement. A proof marker backs a slug only in a file matching the project's `invariants.testGlobs` (`**/*_test.go` here, so backing means an executed test line); when `testGlobs` is empty the checker falls back to source-glob scope (`invariants.sources`), so the test-scoped teeth are opt-in. A separate advisory `touches-invariant: <slug> - <note>` marker records a production site that relates to an invariant; it never backs a slug and never fails `awf check`. The checker is language-agnostic (anchored path globs matched against slash-separated repo-relative paths per ADR-0077, e.g. `**/*.go` for any depth or `internal/**` to scope a subtree, plus a literal marker) and enforces symmetrically: for an Implemented ADR, a backed slug with no proof marker in scope, an `unbacked-invariant:` slug that nonetheless carries a proof marker, and an `unbacked-invariant:` slug missing its `Verify:` note each fail `awf check`; a marker naming an undeclared slug and a note-less `touches-invariant:` marker are non-failing advisories. Backing is opt-in per bullet; untagged bullets remain textual contracts. Enforcement has a retirement escape (ADR-0031): a successor Implemented ADR can declare `retires_invariants: [<slug>]` in its frontmatter to drop a slug from enforcement when the change removes its backing code, leaving the declaring ADR's record intact; a retirement naming a slug no Implemented ADR declares is a dangling retirement and fails `awf check`. ADR-0104 exposes the one-to-one Implemented-ADR `slug → declaring-ADR` map (duplicate-refusing, retirement-applying) as `invariants.DeclaringADRs`, shared by `Check` and `awf context`'s Tier-1 relevance join, which ADR-0106 makes a two-marker union scan over `invariants.sources` and `invariants.testGlobs` (recognising both the proof `invariant:` and the `touches-invariant:` marker), labelling each governing invariant backed or unbacked.

Backing is a ledger, not a proof (ADR-0114). The scan is a textual line match: it strips leading whitespace and tests whether a line begins with the marker, with no awareness of whether the marker sits inside a test function or relates to any assertion. So the check confirms a proof marker for the slug exists in scope; it does not confirm the backing test actually asserts the invariant, and a marker over a trivial or non-asserting test still passes. Backing is therefore bookkeeping that a test is *declared* to back the slug (the ADR-to-marker ledger `awf check` polices tightly), not evidence the property holds. Confirming that a backing test asserts what it claims is the `code-reviewer`'s job, not the scanner's. This is the invariant-scanner counterpart of the coverage gate's limitation (see "Coverage is not verification" in the testing doc): the coverage floor proves a line runs, the marker scan proves a marker exists, and neither proves an assertion.
