# Plan: Pi Fresh-Session Workflow Handoffs

## Goal

Implement [ADR-0145](../decisions/0145-pi-fresh-session-workflow-handoffs.md) so every workflow checkpoint is durable and visible, while Pi 0.81.1 TUI sessions automatically continue in a parent-linked fresh session through a strict `handoff_session` extension. Non-goals are conversation summarization, non-Pi session replacement, destructive session or memory cleanup, and making Pi replacement transactional beyond the runtime's supported boundary.

## Architecture summary

The implementation uses two checked V2 application batches. The first makes the target-neutral checkpoint protocol independently true and moves ADR-0145 to Implementing. The second is intentionally coupled: it adds a closed Pi handoff capability and third target output, implements the separate generated extension and its test seams, raises the shared Pi minimum and pinned runtime, adds Pi-only workflow dispatch, generalizes the container and example wiring, completes documentation, applies the remaining operations, and co-flips ADR-0145 and this plan to Implemented. `internal/project.Target` remains the output-plan authority; the new extension keeps one in-memory pending request and uses Pi's registered-command queue to cross the settled-agent boundary before replacement.

- [ ] **Task 2.3: Implement the generated handoff extension through an injected dependency seam.** Create `templates/pi/awf-handoff/index.ts.tmpl` with the provenance banner supplied by rendering and `// @ts-nocheck` immediately after it. Use injected `lstat`, path resolution, timer, package-version, and extension-file operations; the default factory supplies Node/Pi implementations.

  Register private command name `awf-handoff-continue` before tool `handoff_session`. The public TypeBox schema is exactly:

  ```ts
  Type.Object({
    memoryPath: Type.String(),
    kickoff: Type.String({ maxLength: 1000 }),
  }, { additionalProperties: false })
  ```

  `validateMemoryPath` rejects absolute or backslash paths, empty/noncanonical components, `.` or `..`, `.awf/memory` itself, any path not strictly below `.awf/memory/`, missing components, every intermediate or final symlink, a directory/non-regular final entry, and a resolved path outside the repository derived from the extension location. It reads no file content. `kickoff.trim()` must be nonempty and JavaScript `kickoff.length` must be at most 1,000 UTF-16 code units in addition to schema validation. `buildKickoffWrapper` names the exact accepted path and kickoff and contains the literal instructions to read memory first and treat repository sources/current-state documentation as authoritative.

  The `tool_call` preflight reuses the current-leaf assistant correlation pattern from the subagent extension. When a trustworthy batch contains `handoff_session` and any sibling, block every call in that batch; when correlation is stale/malformed, block the handoff call rather than assuming exclusivity. Before queueing, require `ctx.mode === "tui"`, `ctx.sessionManager.isPersisted()`, and a nonempty session file. Reject print, JSON, RPC, ephemeral, and in-memory sessions without state changes.

  Keep one closure-local pending object `{id, memoryPath, kickoff}`. A second call while pending rejects. After all validation, store a cryptographically unique correlation ID, call `pi.queueCommand("awf-handoff-continue", id)`, clear only that request if queueing throws, and otherwise return a tool result with `terminate: true`. The command rejects a missing or mismatched ID. It consumes its matching request exactly once on success, cancellation, or failure and cannot be replayed.

  Implement the countdown with `ctx.ui.custom`: render exactly `Handoff to a fresh session in Ns - Esc/Ctrl+C to cancel`, decrement once per second from 5, call `tui.requestRender()`, match `tui.select.cancel` through the supplied keybindings manager, proceed on timeout, and clear every interval/timeout on resolve, invalidation, or error. Explicit cancellation notifies `Fresh-session handoff canceled.` and returns with the old session active. Re-run the complete path validation after the countdown and before replacement.

  Call `ctx.newSession({parentSession: oldSessionFile, withSession})` only after revalidation. Inside `withSession`, submit the wrapper only through `replacementCtx.sendUserMessage`. On kickoff rejection, keep the new session active, call `replacementCtx.ui.setEditorText(wrapper)`, and notify `Automatic kickoff failed; submit the prepared editor text.` as a warning. Never reuse the old context after replacement, never delete sessions or memory, and do not promise rollback after teardown begins.

## Notes

- Phase 2 is intentionally large because the old exact-output, editor-strip, and minimum-runtime invariants cannot remain true across a partial third-output rollout. Do not create an intermediate commit that weakens or temporarily removes their proofs.
- `pi.queueCommand` comes from upstream Pi commit `50d88261`; awf consumes the general supported API and introduces no handoff-specific Pi primitive.
- If package version 0.81.1 cannot be resolved reproducibly by the pinned container lane, stop before Phase 2 rather than substituting an unpinned local path or weakening the minimum-version decision.
