# Plan: Supersession Retrofit C: The Citation Check

## Goal

Build and enable ADR-0131 Decision 2's citation check: add the `cites:` token to the grammar,
tokenize the corpus's informational citations, extract citations as a `Corpus` method, land the
check in a new `internal/project/citations.go` wired into `Check()`, back all twelve of ADR-0131's
declared invariants, sweep the doc-currency surfaces, and flip ADR-0131 to `Implemented`.

Non-goals: the relation corrections and slug retirements (Plan A) and the token backfill (Plan B).
**Both must be fully landed before this plan starts**, or enabling the check reds this repo's own
gate on the first run.

## Architecture summary

Three phases, ordered by two hard constraints.

**The dead-code gate (ADR-0063) forbids splitting the build.** `cmd/deadcodecheck` fails on any
production function unreachable from a `main`, with no ignore directive. So the extraction method,
the check, and its wiring into `Check()` land in one phase: an extraction method whose only caller
arrives next phase is dead code, and so is an unwired check.

**The check must not fire on landing.** Phase 3 turns it on, so every legitimate citation must
already carry its token. Phase 1 teaches the parser `cites:`, Phase 2 applies it corpus-wide, and
only then does Phase 3 wire the check. `cites:` tokens inserted before Phase 1 would be inert prose,
harmless but unverifiable, so Phase 1 goes first.

**Placement is forced by ADR-0130.** `corpus-owns-field-reads` forbids any file outside
`internal/adr` from reading `ADR.Refs` or `Sections`, and that invariant is clean today
(`internal/adr/corpus_test.go:123`). Citation extraction is therefore a `Corpus` method;
`internal/project` consumes parsed citations only and never touches raw section text.

One hazard, called out because missing it silently destroys the corpus: **the code-span exemption
must not be shared with `parseRefs`.** ADR-0131 Decision 5 exempts citations and verbs inside inline
code spans, but relation tokens are *themselves* always written inside backticks
(`internal/adr/adr.go:81-85`). A code-span stripper applied to token recognition would delete every
token in the corpus, and the resulting `awf check` output would look like a clean corpus rather than
a bug. The exemption belongs to the citation scanner alone.

## Phase 1: The `cites:` token

- [ ] **Task 1.1: Add `cites:` to the inline token grammar.** In `internal/adr/adr.go`, add a
  pattern beside the existing three (lines 81-85):

  ```go
  	// citesItemRe / citesInvRe mark a citation informational (ADR-0131 item 4).
  	// Inert: parsed so the citation check can see them, counted toward no
  	// anchor's coverage, rendered nowhere. Two patterns, not one alternation,
  	// for the reason itemRefRe and invRefRe are separate - the kind is named by
  	// the pattern, never inferred from the anchor (ADR-0120 item 1).
  	citesItemRe = regexp.MustCompile("`cites: ADR-([0-9]{4})#([1-9][0-9]*)`")
  	citesInvRe  = regexp.MustCompile("`cites-invariant: ADR-([0-9]{4})#([a-z0-9-]+)`")
  ```

  **Do not collapse these into one pattern with an alternation.** `[a-z0-9-]+` already matches
  digits and Go's alternation is leftmost-first, so in `([a-z0-9-]+|[1-9][0-9]*)` the second branch
  is unreachable and every anchor captures as a slug. The shape is not recoverable from the captured
  text in general either: the slug grammar admits an all-digit slug, which is exactly why
  `internal/project/supersession.go:132` keys anchors by kind prefix and notes that an item ref `#2`
  and a slug ref `#2` into one target are distinct anchors.

  **Amended during execution: the slug key is `cites-invariant:`, not `cites:`.** This plan's first
  draft gave both patterns the same `cites:` key, differing only in anchor shape. That double-
  matches: `[a-z0-9-]+` matches digits, so a real item-anchored token such as
  `` `cites: ADR-0050#3` `` matches *both* patterns and `parseRefs` emits two refs, the second a
  phantom slug anchor `3`. `supersession.go:168` then checks that slug against the target's declared
  slugs, finds nothing, and raises `adr-token-ref` drift. All 21 citation tokens in the corpus are
  item-anchored, so Phase 1 would have failed its own gate 8 times on landing and 21 times after
  Phase 2. Two keys make the kind explicit, exactly as `supersedes:`/`supersedes-invariant:` already
  do (ADR-0120 item 1). ADR-0131 Decision 4 was amended to match while still `Proposed`.

  Add a third `Relation` constant beside `Retires` and `Refines` (lines 70-75):

  ```go
  	// Cites is `cites:`: an informational citation asserting no claim on the
  	// anchor. It contributes to nothing and exists only to suppress the
  	// citation check (ADR-0131 item 4).
  	Cites Relation = "cites"
  ```

  In `parseRefs`, call the existing `collect` closure twice, unchanged:
  `collect(citesItemRe, Cites, false)` and `collect(citesInvRe, Cites, true)`. No new argument and
  no shape detection. Update the `Relation` doc comment (lines 58-67) to name all three relations.

- [ ] **Task 1.2: Keep `cites:` out of every derived count.** Read `buildCoverage` in
  `internal/adr/coverage.go` and confirm it contributes only on `adr.Retires`. If it filters
  negatively (anything that is not `Refines`), change it to test `Retires` explicitly: a negative
  filter would count `cites:` as a retirement and kill live ADRs.

  In `internal/project/supersession.go`, `computeSupersession` already gates the contest claim on
  `r.Relation == adr.Retires` (line 152), so `cites:` cannot contest. Its token-ref validity,
  back-pointer, and `Proposed`-target checks apply to every ref regardless of relation, which is
  intended: a `cites:` token names a real anchor and still owes a back-pointer.

- [ ] **Task 1.3: Prove the two provable `cites:` invariants.** In `internal/adr/adr_test.go` and
  `internal/adr/corpus_test.go`, add tests carrying these proof markers:

  ```go
  // invariant: cites-token-uncounted
  // invariant: cites-token-unrendered
  ```

  `cites-token-uncounted`: an ADR whose every anchor carries only `cites:` tokens from `Implemented`
  ADRs is **not** `StateCovered`, and its status stays live.
  `cites-token-unrendered`: rendering ACTIVE.md over a corpus containing a `cites:` token produces
  no supersedence annotation for that anchor, and `awf context` likewise.

  One of the two tests must drive a **slug-anchored** token (`cites-invariant: ADR-NNNN#<slug>`) through
  `citesInvRe`. No slug-anchored `cites:` exists anywhere in the corpus or in Phase 2's insertions,
  all of which are item-anchored, so without a synthetic one that pattern is executed but never
  behaviourally exercised: it would match nothing real, and a mistake in it would not fail.

  `cites-token-suppresses-citation-check` cannot be proved until the check exists; it lands in
  Phase 3 with the other check invariants.

- [ ] **Task 1.4: Verify and commit.** Parsing is **not** unchanged in practice: ADR-0131 already

## Phase 3: The check, its proofs, the doc sweep, and the flip

One commit, principally because the dead-code gate forbids landing the extraction method or the
