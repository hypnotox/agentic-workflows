# Plan: Pi Dashboard Deterministic Lifecycle and Pinned Runtime

## Goal

Implement [ADR-0149](../decisions/0149-deterministic-effort-lifecycle-accounting-and-pi-workflow-routing.md) and [ADR-0150](../decisions/0150-git-ref-pinned-development-runtime.md) so Pi workflow selection deterministically owns effort lifecycle and complete footer-parity accounting, while the from-source repository uses one immutable Git-ref-pinned canonical query runtime. Non-goals are protocol-1 migration, inferred effort identity, dynamic runtime skill rendering, preventing arbitrary same-user filesystem access, hot-swapping existing Pi sessions, subscription or automatic-context display through private Pi APIs, and cache garbage collection.

## Architecture summary

The implementation first adds an isolated pinned-runtime package and a closed private read-only CLI dispatch, then upgrades the telemetry descriptor and Go projector to protocol 2 in lockstep with generated TypeScript. Pi rendering changes from individually discovered governed skills to one router plus hidden pre-rendered bodies, with catalog-owned lifecycle metadata driving `awf_workflow`. The dashboard then adopts provisional association, structured resume, generation-ordered canonical refresh, finding-owned mutations, and Pi-native usage accounting. Each ADR's declared operations apply in declaration order with exactly the claim mutations made true by that batch; both ADRs and this plan freeze only after the complete behavior and documentation pass implementation review.
