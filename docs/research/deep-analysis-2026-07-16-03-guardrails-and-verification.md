# awf Deep Analysis 2026-07-16 - Guardrails and Verification

*Part 03 of the 2026-07-16 analysis set (companion to `deep-analysis-2026-07-16-00-overview.md`). Topic: the deterministic guardrail system and how honestly awf reports what it actually verifies. Delta on `deep-analysis-2026-07-15.md` and `agentic-workflow-landscape-and-awf-standing-2026-07.md`.*

## In one paragraph

The deterministic layer is strong mechanical engineering and it verifies a lot for real: the hermetic staged-slice pre-commit genuinely closes the partial-staging build/drift hole the 07-15 report praised, covercheck and its Codecov filter share one ignore list so the gate and the report cannot disagree, releasecheck re-derives changelog order because release CI runs no tests, and every trust-bearing write is atomic behind a single corrupt-lock refusal. The honest gaps are all at the seam between what the slice checks and what it delegates: the slice only build- and drift-checks the staged bytes, so every behavioral gate (test, coverage, vet, lint, deadcode, prose-gate, pincheck) still runs against the worktree and is blind to partial staging - the exact class the slice exists to close. prose-gate (new since 07-15) reads worktree content while enumerating index paths, silently skips any non-UTF-8 file, and does not fail on a stale exemption; pincheck's line scan is bypassable with two ordinary valid YAML forms. On verification honesty, coverage stays a liveness metric: the largest and fastest-growing coverage-ignore bucket encodes unverified call-order claims, mutation testing is still manual and advisory, and the invariant ledger is a bookkeeping ledger not a proof (now honestly repositioned by ADR-0114). On process, the ADR machinery raised its craft after the 07-15 critique but not its admission bar: the append-only guarantee took three carve-outs in one week, decision half-life is measured in hours, and the warrant line still says "When in doubt, write the ADR."

## What is genuinely good

The mechanical guardrails are the most defensible part of the system, and this holds under this week's ~5-ADR/day velocity.

- **The hermetic staged-slice pre-commit is still the single best guardrail.** `.githooks/pre-commit` materializes the exact staged bytes with `git checkout-index --prefix` into a throwaway dir, then runs `go build ./...` and `./awf-slice check` against that slice (`.githooks/pre-commit:13-22`). It cites the real incidents it would have caught (the ADR-0088 non-building slice, a85bd6a's drifted-but-clean-worktree commit) and, as of ff7d9e2b, drift-checks `examples/sundial` in the same slice (`.githooks/pre-commit:26-28`). This is a real advance over a worktree-only check and the comment's rationale is accurate.
- **The drift oracle cannot silently disagree with itself.** covercheck merges the per-test-binary duplication of a `-coverpkg` profile (`internal/coverage/coverage.go` `mergeBlocks`) and shares `ignoredLines` verbatim between the 100% `Check` and the Codecov `Filter`, so the gate's "100%" and the report's "covered" set can never diverge on what is excluded.
- **Defense-in-depth where CI runs no tests.** releasecheck re-checks changelog ordering itself (`cmd/releasecheck/main.go`) precisely because release CI runs no test suite, so a mis-sorted file cannot pass merely because `entries[0]` matched. pincheck closes the subtle "goreleaser-action step with no `version:` input floats to latest" hole, not just the floated-range case (`cmd/pincheck/main.go:67-76,95-101`).
- **The 100% gate stays honest at the entrypoints.** The `os.Exit`-wrapper-plus-tested-`run()` seam is applied consistently across covercheck, deadcodecheck, pincheck, releasecheck, and repoaudit, so the gate reaches real logic without contorting `main`.
- **Hooks are actually wired.** `core.hooksPath=.githooks`, the stubs delegate to the drift-checked `.awf/hooks` payloads, so the artifact that is checked is the artifact that runs. awf never activates hooks in an adopter tree; it wires its own by hand, which is the honest model.
- **prose-gate has a solid default-deny core.** Exemptions validate the codepoint spelling and reject any rune outside the seven banned substitutes (`internal/prosegate/prosegate.go:53-66`), so a typo cannot silently widen an exemption, and the path set is deliberately the index (a file added in the very commit is in scope).
- **Test assertions reach behavior, not just execution.** The suite favors exact-detail and negative-case checks over liveness: renders are asserted byte-for-byte, the ADR-0076 corrupt-lock matrix asserts zero `.awf-bak` residue and a byte-identical lock after a failed write, and anti-vacuity seen-count guards catch a matcher that would silently match nothing. It runs testify-free across ~126 files (~1050 tests), and the sampled backings turned up no tautological test. This is the substrate that makes the invariant ledger (Finding 5) mostly honest rather than a coverage costume.

The 07-15 verdict that "the mechanical guardrail layer is strong engineering with few escape hatches" still stands. The findings below are the escape hatches, and they cluster at one architectural seam.

## Finding 1 - The staged-slice guarantee is half-open: behavioral gates run on the worktree (CONFIRMED, risk)

`.githooks/pre-commit:14,19` run only `go build ./...` and `./awf-slice check` against the hermetic slice. Line 30 then `exec bash .awf/hooks/pre-commit.sh`, whose body (`.awf/hooks/pre-commit.sh:7-9`) runs `./x check`, `./x gate`, `./x prose-gate` from the original worktree. `./x gate` compiles and runs `go test ./...` over worktree files, so coverage, tests, vet, lint, deadcode, prose-gate, and pincheck all validate the worktree, not the staged bytes. The slice's own comment (`.githooks/pre-commit:3-9`) says it exists because "the payload below validates the worktree", so "a cross-session or partial-staging accident can therefore land a commit that, on its own, does not BUILD ... or is DRIFT-inconsistent" - yet the behavioral gates it hands off to remain worktree-scoped. So the class the slice targets is only half-closed: build and drift are per-commit-hermetic; everything behavioral is not.

**Fix.** Run at least prose-gate and pincheck (both cheap, no test compilation) inside the hermetic slice, or document explicitly that only build+drift are per-commit-hermetic so the guarantee is not overstated. A lightweight pre-push loop that gates each pushed commit would close the intermediate-commit gap that CI (which gates only the pushed tip) leaves open.

## Finding 2 - prose-gate: three reproducible holes in the only blocking repo-wide punctuation guard

prose-gate is new since 07-15 (ADR-0119) and is the sole *blocking* repo-wide punctuation check; the net-increase audit rule in `internal/audit` is advisory only, so nothing backstops these holes. All three were reproduced by the source analyst; I ground-truthed the code paths.

**2a - Reads worktree content while enumerating index paths (CONFIRMED, risk).** `runProseGate` calls `git.IndexPaths(root)` (`cmd/awf/prosegate.go:32`), then `prosegate.Scan` reads each path with `os.ReadFile(filepath.Join(root, p))` (`internal/prosegate/prosegate.go:79`) - the worktree copy. The path set is staged; the content is unstaged. `git add` a file containing an em-dash, overwrite the worktree copy with a hyphen version without re-staging, and prose-gate reports clean while the staged blob still carries U+2014. Because prose-gate is also not in the hermetic slice (Finding 1), nothing catches it. **Fix:** when acting as a commit gate, read the staged blob (`git show :path`) so the path set and the content set come from the same tree.

**2b - Silently skips any non-UTF-8 file (CONFIRMED, risk).** `Scan` does `if !utf8.Valid(b) { continue }` (`internal/prosegate/prosegate.go:83-85`), skipping the whole file. A tracked file `hello <em-dash> world` is correctly flagged; append one invalid byte (0x80) and prose-gate reports clean though the em-dash is still present and committed. `AGENTS.md:45` claims prose-gate "scans every tracked file"; a non-UTF-8 file is a silent third category outside both the scan and the exemption ledger. Note the sibling gate does not share this hole: ADR-0115's shipped-surface scan (`internal/project/residue_scan_test.go`) uses range-over-string, which decodes invalid bytes to U+FFFD rather than skipping. **Fix:** treat a tracked non-UTF-8 file as a finding (or scan its lossily-decoded bytes) unless it is on an explicit binary allowlist.

**2c - Exemptions do not fail when stale (ANALYST-OPINION, smell).** `Scan` iterates only over runes actually present (`for r, n := range counts`, `internal/prosegate/prosegate.go:92`); an exemption pinning a count for a codepoint the file no longer contains is never examined, so a frozen `count: 7` can drift to zero-in-reality unnoticed. This contradicts awf's own fails-when-stale exemption doctrine elsewhere (`internal/project/residue_scan_test.go` errors on an identity-exemption entry the template no longer needs). This is a single-analyst read, not adversarially verified, but the code path is real: I confirmed the loop only visits present runes. **Fix:** report an exemption whose pinned codepoint is absent from its file, matching the residue/identity maps that break the build when stale.

**Adjacent opportunity (ANALYST-OPINION).** `internal/prosegate/prosegate.go:20-22` keeps `Banned` in agreement with `residue_scan_test.go`'s `bannedRunes` by comment only; no test asserts the two blocklists are identical. An 8th banned codepoint added to one gate but not the other would silently split the shipped-surface ban (ADR-0115) from the repo-wide ban (ADR-0119). A one-line equality assertion closes it.

## Finding 3 - pincheck's line scan is bypassable with ordinary valid YAML (CONFIRMED, risk)

`checkFile` matches only `strings.HasPrefix(ln, "uses:")` after trimming and stripping a leading `- ` (`cmd/pincheck/main.go:82-84`). Two valid YAML forms that GitHub Actions accepts slip past:

- Quoted key: `- "uses": evil/action@main` - after strip, `ln` is `"uses": ...`, so `HasPrefix(ln, "uses:")` is false.
- Flow mapping: `- { uses: evil/action@main }` - `ln` is `{ uses: ... }`, again no prefix match.

Both yield "all workflow references pinned" (exit 0), letting an unpinned moving-tag action into CI. pincheck is the supply-chain pin gate (ADR-0079) and runs in `./x gate`, so this is the one guardrail whose bypass injects unreviewed third-party code. The code documents the line-based choice as intentional for "flat enough" YAML (`cmd/pincheck/main.go:61-63`); that assumption is exactly what an adversarial edit breaks. **Fix:** parse the workflow YAML (or normalize quoted keys and flow mappings) before the `uses:`/`version:` scan.

*Lower-severity, ANALYST-OPINION.* `isExemptSubject` in `cmd/awf/commitgate.go` prefix-matches `Merge `, `fixup!`, `squash!`, `amend!`, so an authored subject like `Merge unrelated feature without a scope` bypasses the conventional-commit/scope check entirely. Commit-hygiene only, and only when the commit-msg hook is wired, but it is a deterministic check with a trivially reachable off-switch. Tighten to git's actual generated forms.

## Concrete gate-bypass scenarios

Four adversarial or partial-staging paths, with whether the full gate (`./x gate` / `./x gate full`) catches them at commit time. All rest on CONFIRMED findings above.

| # | Scenario | Caught by staged-slice? | Caught by full gate at commit? |
|---|----------|-------------------------|--------------------------------|
| A | Stage a new uncovered production function; leave its only test in the unstaged worktree | No (slice builds; drift clean) | No - `./x gate` runs `go test` over the worktree, which has the test, so coverage reads 100%. The intermediate commit ships an uncovered function; only a checkout of that commit alone fails. CI gates the pushed tip, not intermediates. |
| B | `git add` a file with U+2014, then overwrite the worktree copy with a hyphen version without re-staging | No (prose-gate not in slice) | No - prose-gate reads the worktree hyphen copy (Finding 2a) and reports clean; the em-dash is committed in the staged blob and stays invisible because the post-commit worktree still shows the hyphen. |
| C | Add a workflow step as `- "uses": vendor/action@main` (quoted key) or `- { uses: vendor/action@main }` (flow mapping) | No (pincheck not in slice) | No - pincheck's line scan (Finding 3) does not match; `./x gate` reports all references pinned; the unpinned action reaches CI. |
| D | Commit a mostly-text tracked file containing U+2014 plus one 0x80 byte | No | No - `utf8.Valid` is false, prose-gate skips the whole file (Finding 2b), the banned glyph ships. |

The through-line: the full gate catches none of these, because its behavioral checks read the worktree and (for B, C, D) carry the content-source or input-form holes above. The staged-slice's hermeticity does not extend to them. Scenarios B, C, and D are also *permanent* misses, not just deferred - the banned glyph or unpinned action stays in history and the gate never re-examines it on a clean worktree.

## Finding 4 - Coverage-ignore: the sequencing-claim bucket is the largest and encodes unverified call-order claims (CONFIRMED, risk)

Live census on the current tree: **192 `// coverage-ignore:` directives (190 production, 2 in `_test.go`).** The largest single semantic bucket is not OS-fault unreachability but cross-function call-order claims - "X above already read this path", "generateActiveMD above parses the same dir and fails first", "planSections above already read every part". By one regex this bucket is ~49 of the 190 production directives (the source analyst counted ~56 with a broader match); the permission/root-bypass bucket is ~43. Either way, sequencing claims lead, and they are the fastest-growing category across the multi-ADR waves.

The canonical example the source finding cited as `render.go:605` is actually at **`internal/project/render.go:642`** and its twin **`internal/project/project.go:140`** (both read "unreachable - generateActiveMD above parses the same decisions dir and fails first on a malformed ADR"); line 605 is `return out, nil`. I corrected the citation. The `internal/migrate/retirementtokens.go` ADR-0120 retrofit added "ParseDir above already read this exact path" claims of the same shape (I count two there).

Why this is a risk and not a nit: these directives encode a claim about *another function's* behavior. A refactor that makes the pre-empting read lazy, moves it, or changes its error handling can make the ignored branch reachable while the marker line itself is untouched - and the only guard, the repoaudit `coverage-ignore-added` advisory (`cmd/repoaudit/main.go`), is Warning-only and fires solely when the diff touches the marker line. The dangerous case is exactly the unguarded one. Coverage stays, in the 07-15 report's phrase, "a liveness metric wearing a verification costume", and the biggest chunk of that costume is now these call-order assertions.

**Fix (deterministic, cheap, never shipped to adopters).** Add a periodic ignore-revalidation pass: strip all directives, run the suite once, diff the uncovered-line set against directive lines. Any directive whose branch the suite now reaches gets re-flagged. This is the fails-when-stale idiom awf already uses everywhere, applied to its own ignore ledger, and it costs one extra covercheck run.

*Two inert directives (CONFIRMED, nit).* `internal/project/glossary_test.go:29` and `internal/project/pitfalls_test.go:29` both carry `// coverage-ignore: t.Fatal never returns` inside `_test.go` helpers. Go never instruments `_test.go` statements, so covercheck can never see these lines - the markers are decoration and mildly inflate the census. Delete both, and optionally teach the repoaudit rule to error on a directive in a `*_test.go` file.

## Finding 5 - Invariant backing is a ledger not a proof; awf's own backings are mostly honest (mixed)

ADR-0114 (landed 07-15) repositioned invariant "backing" from "test-proven" to "declared-to-back" on four shipped surfaces and charged the code-reviewer with the semantic check - the 07-15 T1 recommendation, executed same-day (verified live at `.claude/agents/code-reviewer.md:47`). This delta assesses whether awf's *own* invariant tests actually assert what they back, now that the ledger's honesty is the reviewer's job.

The answer is mostly yes. Across a 19-slug sample the source analyst found 15 backings sitting on tests that genuinely assert the declared claim, several with exact-detail and negative-case assertions; the ledger symmetry holds mechanically today (`awf invariants` clean, ~321 backed declarations, exactly one `unbacked-invariant` with its Verify note, zero dangling markers). The weakest sampled entries (ANALYST-OPINION):

- **Three ADR-0090 invariants are backed by substring scans of the `./x` script.** `example-adopter-checked`, `example-zero-notes`, `example-module-isolated` are all backed by `TestExampleAdopterWiring` (`internal/project/example_wiring_test.go`), which greps `../../x` for literals like `grep -q '^note: '`. ADR-0090 declares "the example check step fails on any `note:` line", but the test asserts only that the grep *text* exists in the script; the surrounding shell logic could invert or disconnect it and the pin would still pass. This is the furthest the sampled ledger drifts from behavior. Fix: extract the note-failing pipeline into a form the test can run against a fixture producing a `note:` line, or annotate these markers as deliberate script-text proxies.
- **`gitfixture.Commit`'s ignore reason is factually wrong (nit).** `internal/testsupport/gitfixture/gitfixture.go:45` claims the write-error branch "fails only on a permission fault a test cannot trigger", but `Commit` calls `os.WriteFile` with no `MkdirAll`, so a nested name like `sub/f.txt` hits ENOENT and reaches the branch. Behavior is fine (it `t.Fatal`s); only the unreachability claim is false.

Structurally, the deeper point from 07-15 stands: mutation testing - the one tool that would catch "100% coverage with trivial assertions" - remains manual and advisory, absent from `./x gate` and CI. The recipe is known and recorded (`-i --workers 1 --timeout-coefficient 20`, trust only `Timed out: 0`), and ADR-0114 explicitly names scoped nightly mutation runs against invariant-guarded paths as a "separate, already-tracked effort" that nothing in this delta advanced. Pointing `cmd/mutants` at the enclosing functions of the ~49 sequencing-claim ignore sites first would put the signal exactly where the 100% number says least.

## Finding 6 - ADR corpus and process health: craft rose, admission did not

The corpus is mechanically healthier than it has ever been. ADR-0120 turned supersession - the weakest relation - into a symmetric machine-checked one: 3 Superseded ADRs with matching frontmatter/status/ACTIVE.md chains, 15 `supersedes-invariant` token anchors all rendered, the 10 back-pointer edges ADR-0116 measured as missing now backfilled, `awf check` and `awf invariants` clean, and the ADR-0121 `close:` token implemented at both scan seams with a proof-marked test. Individual ADR craft visibly rose after the 07-15 critique: 0116 measures its own corpus before deciding, 0119 documents a defect in its own predecessor chain's measurement, 0120's alternatives table labels a rejected option "the strongest rejected option, and not refuted." That is real self-honesty.

The process-level half of the 07-15 critique did not land.

**6a - The append-only guarantee took three carve-outs in one week (CONFIRMED, risk).** ADR-0116 Decision 2 (07-15) permits status-plus-cross-reference edits; ADR-0118 (07-15) permits meaning-preserving orthography sweeps; ADR-0120 Decision 9 (`docs/decisions/0120-structured-machine-checked-adr-supersession.md:152-158`, 07-16) permits three enumerated encoding-edit shapes and restates the rule as "the body's meaning is frozen." 0120's own Consequences concede it: "The append-only rule is no longer absolute" and "reviewers must now distinguish encoding edits from content edits when a diff touches a frozen body" (`0120-...md:242-244`). The legal-edit set for the corpus's central invariant now spans three ADRs and rests on the elastic judgment "meaning-preserving"; the 0120 retrofit already hand-edited dozens of frozen bodies under it. Each carve-out is individually well-reasoned; the accumulation - one new mutation class per justified need, weekly - is the erosion vector. **Fix:** consolidate the permitted-edit taxonomy in one place a reviewer actually loads (the adr-lifecycle skill), and treat a fourth carve-out proposal as the trigger for a hard-ceiling decision.

**6b - The warrant bar was not raised (ANALYST-OPINION, smell).** The 07-15 T2-7 asked for a scope-ceiling "what awf is not" ADR and a raised warrant bar ("what will the project forget without this? - 'nothing' is a valid answer"). Neither exists: no scope-ceiling ADR appears in `docs/decisions/`, and both the rendered skill and its template still end with the write-more default - `.claude/skills/awf-proposing-adr/SKILL.md:18` and `templates/skills/proposing-adr/SKILL.md.tmpl:18` read "When in doubt, write the ADR." Meanwhile the recommendations that *produce* ADRs (0112 trim, 0114 positioning) landed within 24h. The asymmetry is structural: the machinery makes additive ADRs cheap and restraint ADRs unrepresented, so the accretion-curbing recs are precisely the unexecuted ones.

**6c - Decision half-life measured in hours (ANALYST-OPINION, smell).** 5 of the 10 delta ADRs were wholly or partially overridden inside the same 48h window. ADR-0113 (`docs/decisions/0113-em-dash-free-shipped-templates.md`) was Implemented and Superseded the same day; its Alternatives row rejects "ban all machine-set punctuation" as something that "would fail on landing and demand an unrequested cleanup", and ADR-0115 adopted exactly that hours later. The punctuation policy alone consumed 5 ADRs (~14.5k words, ~7% of the corpus) in 48h. Contrast ADR-0120, which absorbed the same escalation pressure correctly with four amendment-while-Proposed commits instead of successor ADRs. The lesson is already demonstrated inside the delta: when a thread is visibly escalating, amend the Proposed ADR rather than open a successor.

**6d - Additive:simplifying ratio (ANALYST-OPINION).** Over roughly ADR-0102..0121 the mix is about 14 additive : 5 simplifying : 1 consolidating - restraint is present (0107, 0108, 0112, 0114, 0116 all trim or decline) but outnumbered ~3:1. This is now mechanically measurable: a repoaudit advisory classifying each new ADR (presence of `supersedes`/`supersedes-invariant` tokens, or a frontmatter tag) would turn the health metric the 07-15 report asked for into one glance per review.

**6e - The standard and its own violation shipped in the same week (ANALYST-OPINION, smell).** ADR-0112's retention criterion trimmed the guide Invariants to 11 bullets but declined the bullet-length lint (Decision 4); within 24h two of those bullets re-bloated past the doc standard's own "one terse imperative line each" rule (`AGENTS.md:44-45` run 636 and 918 chars). The standard-setter fell out of conformance with itself a day after setting the rule - the same self-consistency gap the mechanical gates exist to prevent, but on a surface no gate scans. This is the T2-4 delta row read as a smell rather than a metric. **Fix:** either restore the declined bullet-length lint or split the two multi-clause bullets so the guide models the rule it publishes.

*Nit (ANALYST-OPINION).* `docs/decisions/ACTIVE.md` emits a standing `note: anchor ADR-0034#1 claimed by ADR-0057 and ADR-0121` on every `awf check` run; per ADR-0120 item 5 this clause-level split is legitimate by design, so the note can never be cleared, which trains readers to skip a channel that also carries actionable advisories. Pin a known-legitimate-split acknowledgement (the count-pinned exemption pattern already models this).

## Prioritized list for this topic

1. **Close the staged-slice behavioral gap for the two cheapest, most-adversarial gates.** Run prose-gate and pincheck inside the hermetic slice (Findings 1, 2a, 3). This kills bypass scenarios B and C at commit time and costs no test compilation.
2. **Fix prose-gate's content source and non-UTF-8 skip** (Findings 2a, 2b). Read the staged blob when gating; treat a tracked non-UTF-8 file as a finding. These are CONFIRMED risks in the only blocking punctuation guard.
3. **Harden pincheck against non-line YAML** (Finding 3). Highest blast radius of any guardrail hole: its bypass injects unreviewed CI code.
4. **Add the coverage-ignore revalidation pass** (Finding 4). Deterministic, one extra covercheck run, retires the largest and least-verified ignore bucket's rot mode, and deletes the two inert `_test.go` markers.
5. **Consolidate the append-only carve-out taxonomy and treat a fourth as a ceiling trigger** (Finding 6a). The central invariant's legal-edit set now spans three ADRs on an elastic judgment call.
6. **Raise the ADR warrant bar and write the scope-ceiling ADR** (Finding 6b). The unexecuted half of the 07-15 critique, and the reason the corpus grows faster than it prunes.
7. **Lower-priority hardening:** assert `Banned` == `bannedRunes` equality, make prose-gate exemptions fail-when-stale, convert the ADR-0090 script-pin invariants to executed behavior, and add a repoaudit additive:simplifying classifier for the health metric.
