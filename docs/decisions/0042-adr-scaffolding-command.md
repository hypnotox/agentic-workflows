# ADR-0042: `awf new adr` Scaffolding Command

## Context

Proposing a new ADR (`awf-proposing-adr`) currently requires an agent to do three mechanical
things by hand: list `docs/decisions/NNNN-*.md` and take the max number plus one, copy the section
structure out of the rendered `docs/decisions/template.md`, and strip that file's
`<!-- GENERATED by awf -->` / `<!-- awf:edit ... -->` marker comments before the content becomes a
real ADR. None of this is judgment — it is the same three steps every time — but doing it free-hand
risks a stray marker comment surviving into a committed ADR, or a numbering mistake from misreading
the directory listing.

We want this specific mechanical step, and only this step, moved into the `awf` binary. Authoring
the actual ADR content (Context/Decision/Invariants/Consequences/Alternatives, and the frontmatter
`tags`/`related`/`domains` arrays) stays exactly where it is today: in the agent following
`awf-proposing-adr`.

The next available number today is `0042` (this ADR).

Domains and skills are not a second instance of this problem: they are catalog-toggled via the
existing `awf add <kind> <name>` command (enabling/disabling a fixed, shipped artifact), not
freely-authored prose like an ADR. A generalized `awf new <kind>` framework for those is explicitly
deferred until a second concrete, freely-authored-content case exists (see Alternatives Considered).

## Decision

1. `internal/adr` (currently parse/render-only: `ParseDir`, `RenderActiveMD`, `RenderDomainIndex`)
   gains two functions:
   - `NextNumber(dir string) (string, error)` — computes the next zero-padded 4-digit number from
     the max `Number` returned by `ParseDir` (an empty directory yields `"0001"`).
   - `NewFile(dir, title string) (path string, err error)` — reads the rendered
     `docs/decisions/template.md` in `dir`, strips every `<!-- GENERATED by awf -->` and
     `<!-- awf:edit ... -->` marker comment line, sets the frontmatter `date:` to today via a
     package-level `var now = time.Now` seam (mirroring the existing `var getwd = os.Getwd` /
     `var isInteractive = ...` seam pattern in `cmd/awf/main.go`, so the date is deterministic under
     test per the 100%-coverage gate), computes the filename as `NNNN-<slug>.md`, and refuses to
     overwrite an existing file at that path.
   - Slugification: lowercase the title, collapse every run of characters that is not `[a-z0-9]`
     into a single `-`, then trim leading/trailing `-`. An empty result (e.g. a title with no
     alphanumeric characters) is a hard error — there is no existing slug helper elsewhere in the
     codebase to defer to, so this is the full algorithm.
   - The package doc comment (`internal/adr/adr.go:1`) is updated to reflect that the package now
     also creates ADR files, not only parses/renders them.
2. `internal/project.Project` gains an exported `NewADR(title string) (string, error)` method that
   resolves the project's ADR directory via its existing private `decisionsDir()` and calls
   `adr.NextNumber` / `adr.NewFile`. This mirrors the existing `CheckInvariants`/`Audit` pattern:
   `cmd/awf` never reaches into `internal/project.Layout` fields directly, only through an exported
   `Project` method.
3. `cmd/awf/new.go`: a new top-level `new` command, added to `commandOrder` and `argSpecs` in
   `cmd/awf/main.go`. Invocation: `awf new <kind> <title...>`. Today only `kind == "adr"` is valid;
   any other value is a usage error (exit 2), mirroring `list_add.go`'s `unknownKind` helper but
   with its own message (`unknown kind %q (want: adr)`) — `add`/`remove`'s kind vocabulary
   (`skill, agent, doc, domain, target, bootstrap`) is a distinct, unrelated space and `adr` is not
   added to it. The command calls the shared `gate(root)` version check (ADR-0039) before
   `project.Open`, exactly like `sync`/`check`/`invariants`/`audit`/`list` already do, since it reads
   the rendered `docs/decisions/template.md` and a version-mismatched binary could misparse its
   marker format. On success it prints the created file's path to stdout.
4. ADR-0039's invariant text ("every gated command") is amended to include `new` in its enumerated
   command list.
5. `awf-proposing-adr`'s procedure (source: `.awf/skills/proposing-adr.yaml`) changes: the current
   step 1 ("pick the next ADR number") and the file-creation half of step 2 ("copy section structure
   ... do not shell-copy the file") collapse into a single step that runs `awf new adr "<title>"`
   and then fills in every section and frontmatter array in the resulting file. The prohibition on
   shell-copying the template is superseded by this command — `awf new adr` *is* the sanctioned
   copy path, not a hand-rolled one.

## Invariants

- `inv: adr-new-strips-markers` — `awf new adr`'s output file never contains a `GENERATED by awf`
  or `awf:edit` marker comment line.
- `inv: adr-new-sequential-numbering` — `internal/adr.NextNumber` always returns the max existing
  ADR number plus one, and never reuses a number already present in the directory.
- `inv: adr-new-no-overwrite` — `awf new adr` refuses to overwrite an existing file at its computed
  target path rather than silently clobbering it.
- `inv: adr-new-version-gated` — `awf new adr` invokes the ADR-0039 binary-version gate before
  reading or writing any project file.
- The `awf-proposing-adr` skill never instructs an agent to hand-copy `docs/decisions/template.md`
  — `awf new adr` is the only sanctioned creation path (textual contract on skill content, not
  machine-checked).

## Consequences

Proposing an ADR becomes one deterministic command plus the same content-authoring work as today —
removing the two failure modes (stray marker comments, numbering mistakes) that motivated this ADR,
without changing who writes Context/Decision/Invariants/Consequences/Alternatives.

`internal/adr` grows a write path (`NewFile`) alongside its existing parse/render responsibilities;
its package doc comment must stay accurate as more write operations are added later, or this ADR's
own scope will have quietly widened the package without documenting it. `cmd/awf/new.go` is a new
top-level command surface with its own test file (`new_test.go`), following the existing
per-command test convention (`invariants_test.go`, `audit_test.go`, etc.).

ADR-0039's gated-command list changes from a fixed five (`sync`, `check`, `invariants`, `audit`,
`list`) to six, so any future command that reads a rendered artifact must now also ask whether it
belongs on that list — the criterion (reads binary-schema-sensitive rendered output) is now
established by precedent, not just named once.

No generalized scaffolding framework exists yet for domains or skills; if a second freely-authored
template case appears, `awf new <kind>` already has a kind-dispatch shape to extend rather than
redesign.

## Alternatives Considered

| Alternative | Why not chosen |
|---|---|
| Generalized `awf new <kind>` framework covering domains/skills now | Domains and skills are already catalog-toggled via `awf add <kind> <name>` (a fixed, shipped artifact), not freely-authored content like an ADR — there is no second concrete use case yet, so a generic template-scaffolding framework would be speculative. |
| A shell script (e.g. `docs/decisions/new-adr.sh`) instead of a Go command | Inconsistent with every other awf operation being a cross-platform Go binary subject to the 100%-statement-coverage gate; a shell script sits outside both. |
| Leave the status quo (agent copies and strips markers by hand) | This is the exact problem motivating the ADR: mechanical, deterministic work currently done free-hand, with real observed risk of stray markers or numbering mistakes. |
