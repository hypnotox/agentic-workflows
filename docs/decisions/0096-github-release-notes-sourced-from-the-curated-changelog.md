# ADR-0096: GitHub Release Notes Sourced from the Curated Changelog

## Context

The GitHub Release notes are what an adopter reads to learn what changed in a version.
They were generated by GoReleaser from git commit messages (`.goreleaser.yaml`,
`changelog.use: github`), with a `filters.exclude` list meant to drop internal noise:
`^chore:`, `^docs:`, `^test:`, `^ci:`.

Those patterns are unscoped, and this repository scopes nearly every commit
(`chore(tooling)`, `docs(plans)`, `refactor(tooling)`, …). Go's RE2 anchors `^chore:`
to "chore" immediately followed by ":", so it does not match `chore(tooling): …`; the
commit falls through to the catch-all "Others" group and ships in the release notes.
There is also no `refactor` exclude at all. The observed symptom: the internal-only
`chore(tooling): add ./x context passthrough` commit (a wrapper in this repo's own `./x`
dev runner, no adopter-facing effect) appeared in the adopter-facing release notes.
Because the excludes match almost nothing, this was never a one-off — effectively every
non-`feat`/`fix` commit had been leaking into every release.

Meanwhile the project already maintains `changelog/CHANGELOG.md`: a hand-curated,
newest-first changelog whose entries are grouped by *actual adopter-facing effect* (does
it change rendered template output, CLI behavior, or config/lock schema), not by mirroring
a commit's Conventional-Commits type. That file is release-gated — `cmd/releasecheck`
(ADR-0078, `inv: release-changelog-pin`) refuses a tag unless the newest entry equals
`project.Version` — and `awf changelog --version <v>` (ADR-0041) already extracts exactly
one version's section. GoReleaser ignored all of this and re-derived, and re-corrupted,
the notes from commit subjects.

Filtering the commit stream better (scope-aware regexes, a `refactor` entry) would patch
the immediate leak, but keeps two sources of truth that can disagree and stays a denylist
that the next new internal commit type slips past. The curated changelog is the authority
on adopter-facing effect; the release notes should present it, not compete with it.

## Decision

1. **The GitHub Release body is the curated changelog section for the tagged version, not
   a commit-derived changelog.** The `.goreleaser.yaml` `changelog` block collapses to
   exactly `changelog:` / `disable: true` — the `use: github`, `sort: asc`, `groups`, and
   `filters` lines are all dropped, not just the exclude patterns. GoReleaser no longer
   derives release notes from commit messages, and the buggy exclude patterns are deleted
   rather than repaired. The residual two-line block is a stable token the gate test can
   match to prove the commit-derived changelog is off.

2. **The release workflow feeds the curated section to GoReleaser.** `.github/workflows/release.yml`
   extracts it after the `releasecheck` step and before the GoReleaser step —
   `go run ./cmd/awf changelog --version "${GITHUB_REF_NAME#v}" > "$RUNNER_TEMP/release-notes.md"` —
   and the GoReleaser action passes `--release-notes` that same absolute path in its `args`.
   The notes file is written **outside the git worktree** (under `$RUNNER_TEMP`): GoReleaser's
   release pipeline aborts on a dirty tree, and an untracked `release-notes.md` at the repo
   root would count as dirty, so the notes must not land in the checkout. The full section
   text is used verbatim, including its leading `## [<version>] - <date>` header.
   `releasecheck` (run earlier in the same workflow) guarantees the newest section equals
   the tagged version, so the extraction always resolves; an absent version fails the step
   non-zero rather than shipping empty notes. The exact `args` interpolation is settled at
   implementation and validated with `goreleaser check` against the pinned v2.17.0.

3. **A gate test backs the wiring.** A test in `cmd/releasecheck/main_test.go`, mirroring
   the existing `TestReleaseWorkflowRunsReleasecheck`, asserts that `release.yml` extracts
   notes via `awf changelog --version` before the GoReleaser step, that the GoReleaser step
   passes `--release-notes`, and that `.goreleaser.yaml` disables the commit-derived
   changelog — so unwiring any of the three fails `./x gate`.

## Invariants

- `invariant: release-notes-from-changelog` — the GitHub Release body is sourced from the curated
  changelog: `.goreleaser.yaml` disables GoReleaser's commit-derived changelog, and a gate
  test asserts `release.yml` extracts the tagged version's section via `awf changelog
  --version` and passes it to the GoReleaser step through `--release-notes`, both before
  that step runs. A commit subject can no longer reach the release notes.

## Consequences

- Release notes now match the curated changelog exactly — one source of truth, grouped by
  adopter-facing effect, and the internal-commit leak class is eliminated by construction
  rather than filtered. This subsumes every scoped Conventional-Commits type at once,
  including the `refactor` type the old denylist never covered.
- The curated changelog becomes a hard release input: a malformed or missing section for
  the tagged version fails the release step. This is already guaranteed one step earlier by
  `releasecheck` (ADR-0078), so the coupling adds no new authoring burden — it makes the
  existing gate load-bearing for the notes too.
- The release body opens with the `## [<version>] - <date>` H2 that GitHub also renders as
  the release title; this mild redundancy is accepted in exchange for using `awf changelog`'s
  output verbatim (no fragile header-stripping in the workflow).
- The notes file is written under `$RUNNER_TEMP`, outside the checkout, precisely because
  GoReleaser's release pipeline refuses to run against a dirty git tree (an untracked file
  counts). Writing it into the repo root would have broken the very release step this ADR
  fixes — the notes path must stay out of the worktree.
- `.goreleaser.yaml` no longer classifies commits into changelog groups. Nothing else in
  the config consumed the changelog (no `announce`/`blob` sections), so no other behavior
  changes.

## Alternatives Considered

| Alternative | Why not chosen |
|---|---|
| Scope-aware exclude regexes (`^chore(\(.+\))??!?:` …) plus a `refactor` entry | Patches the immediate leak but keeps two changelog sources that can disagree, and a denylist still leaks the next internal commit type not enumerated. |
| Allowlist filter — only `^feat`/`^fix` reach the notes | Nothing leaks, but it silently under-reports adopter-facing `refactor(...)` and breaking-only changes. |
| Leave GoReleaser deriving notes, edit the release afterward by hand | Manual, unenforceable, and re-opens the leak on every release. |
