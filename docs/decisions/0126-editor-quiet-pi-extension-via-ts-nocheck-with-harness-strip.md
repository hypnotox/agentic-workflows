# ADR-0126: Editor-Quiet Pi Extension via ts-nocheck with Harness Strip


## Context

ADR-0123 ships two generated TypeScript files, `.pi/extensions/awf-subagents/index.ts`
and `runner.ts`, rendered from `templates/pi/awf-subagents/*.tmpl` and wired into the Pi
target at `internal/project/target.go:187-188`. They use Node globals (`Buffer`,
`process`, `NodeJS.*`) and `node:*` imports.

No `@types/node` is resolvable anywhere above `.pi/`: there is no `node_modules` and no
ancestor `tsconfig.json`. The only `tsconfig.json` and `@types/node` in the repository
live in the sibling harness subtree `tools/pi-extension-test/`, which an editor's TypeScript
language service does not associate with the extension files (it walks up the directory
tree from each file and finds no owning config). As a result, adopter IDEs statically flag
every checkout. IntelliJ was observed reporting `TS2591: Cannot find name 'Buffer'` through
whole-project inspection without the file even being opened. This is unsolicited noise in
every awf-adopting project, which awf is meant to absorb rather than pass on.

The obvious local remedies fail or cost too much:

- A co-located empty or scoping `tsconfig.json` at the extension directory was tested in
  IntelliJ and made the noise worse: IntelliJ type-checks orphan `.ts` files (those outside
  any config's `include`) under default compiler options, which still lack `@types/node`, so
  scoping the files out increases inspection rather than silencing it.
- `.editorconfig` is a formatting standard with no lever over the TypeScript language
  service.
- IntelliJ project settings that could suppress the inspection live under `.idea/`, which is
  per-developer, machine-specific, and not an awf render target.
- Making `@types/node` genuinely resolvable requires a `package.json` plus an `npm install`
  under `.pi/`, imposing Node tooling and a `node_modules` tree on every adopter. This
  contradicts ADR-0123's Go-first local setup, whose Decisions #6 and #7 deliberately keep
  npm and `node_modules` behind the Docker gate.

The maintainer's constraint is explicit: zero editor noise, and no install step imposed on
adopters. Because no config can make `@types/node` resolvable without an install, the only
route that satisfies both is in-file suppression rather than type resolution.

## Decision

1. Both templates (`templates/pi/awf-subagents/index.ts.tmpl` and `runner.ts.tmpl`) emit
   `// @ts-nocheck` as their first line. The renderer injects the `// GENERATED by awf`
   provenance banner ahead of the template body (`internal/project/banner.go`), so each
   rendered file carries the banner on line 1 and `// @ts-nocheck` on line 2. Every rendered
   Pi extension file, awf's own dogfood copy and every adopter's alike, is therefore
   byte-identical and editor-quiet: the directive suppresses the whole-file type inspection
   that produced the noise, in every `tsc`-based editor service, with no `node_modules`
   anywhere. No config var or render flag gates the directive; a flag would make awf's
   rendered bytes diverge from adopters', breaking the dogfood-fidelity requirement that awf
   model exactly what it generates.

2. The container test lane strips the `// @ts-nocheck` line from its ephemeral checkout copy
   before `tsc` runs. The harness already rewrites its throwaway `/workspace/repo` copy
   inside a single `docker exec ... sh -lc '...'` chain (`tools/pi-extension-test/container.sh`
   copies the source, rewrites `package.json`, and symlinks `node_modules`); a directive-strip
   stage that deletes the `// @ts-nocheck` line by pattern from both rendered files is added to
   that chain after the copy and before its `tsc` stage. This changes what ADR-0123 Decision #7's
   static-type-check binds to: the type-check now runs against the stripped copy of the generated
   extension rather than the verbatim dogfood file, though that copy differs from the shipped
   file only by the one non-type-meaning directive line. `refines: ADR-0123#7`. Decision #6's
   gate lane (Docker image, dependency fingerprint, persistent container) is untouched and still
   binds as written; this is an addition to that lane, linked by the `related:` back-pointer, not
   an override. `container.sh` is not part of the Docker image fingerprint (`hash_files()` hashes
   only `Dockerfile`, `docker-entrypoint.sh`, `package.json`, and `package-lock.json`), so the
   change needs no image rebuild.

3. Implementation lands the two template edits, the `container.sh` strip stage, and the backing
   test together, and in the same commits re-renders both dogfood files
   (`.pi/extensions/awf-subagents/*.ts`) and the Sundial example
   (`examples/sundial/.pi/extensions/awf-subagents/*.ts`) via `./x sync`, updates any prose lane
   doc that describes the type-check lane (`docs/testing.md`), and records the change in the
   changelog. The proof marker for `pi-extension-editor-quiet-strip` lives in a `**/*_test.go`
   test (this repository scopes `invariants.testGlobs` to `**/*_test.go`). The final
   Proposed->Implemented status flip runs `./x sync` and stages the regenerated `ACTIVE.md` and
   domain indexes.

## Invariants

- `invariant: pi-extension-editor-quiet-strip`: both rendered Pi extension files carry
  `// @ts-nocheck` on the line immediately after the provenance banner, and the container
  gate strips that exact directive from its ephemeral copy before `tsc` runs. The two halves
  are coupled and neither stands alone: the directive without the strip would silently drop
  the extension's static type-check, and the strip without the directive would restore
  adopter editor noise. Because a missing strip leaves the gate green (see Consequences), the
  coupling cannot be enforced by the lane outcome; the proof marker is a static `**/*_test.go`
  test asserting both halves directly: (a) line 2 of each rendered file is exactly
  `// @ts-nocheck`, and (b) `container.sh` contains the directive-strip stage ahead of its
  `tsc` invocation.

## Consequences

- Every adopter's editor is quiet on the extension with no install, no config, and no
  per-developer IDE setting; the fix travels in the rendered artifact itself.
- awf's dogfood stays byte-identical to what adopters receive, and the render carries no
  target-conditional branch.
- The static type-check still covers the real extension code, minus one directive line with
  no type meaning, so ADR-0123's verification pillar remains intact. A comment line is
  non-executable, so the container's 100% line, branch, and function coverage counts are
  unaffected.
- The design introduces a deliberate coupling across two files: the templates and
  `container.sh` are each incorrect in isolation. A future reader who sees only one half
  could "fix" it and either restore the noise or silently kill the type-check. The failure
  mode is asymmetric and dangerous: removing the strip does not turn the gate red, because
  `@ts-nocheck` makes `tsc` silently skip the file, so the lane stays green while the
  extension's type-check is disabled. The lane outcome therefore cannot enforce the coupling;
  only the static backing test (asserting the directive on line 2 and the strip stage in
  `container.sh`) can. The invariant and this ADR freeze that coupling, and the backing test
  asserts both halves so it is mechanically enforced.
- `@ts-nocheck` disables type-checking of these files for editors and for any ad-hoc `tsc`
  run outside the gate. Contributors who want real diagnostics must go through `./x gate`,
  which strips the directive. Accepted: the gate is the sanctioned type-check.
- The directive is pure ASCII, so the plain-punctuation and prose gates are unaffected.

## Alternatives Considered

| Alternative | Why not chosen |
|---|---|
| Co-located scoping `tsconfig.json` (empty `include`) | Tested in IntelliJ: orphan `.ts` files get default-option type-checking with no `@types/node`, so scoping them out increases noise instead of removing it. |
| `package.json` + `npm install` under `.pi/` | Actually resolves the Node types, but imposes Node tooling and a `node_modules` tree on every adopter, contradicting ADR-0123's Go-first local setup. |
| `.editorconfig` or `.idea` settings | `.editorconfig` has no control over the TypeScript language service; `.idea` is per-developer and not an awf render target. |
| Render flag: pragma for adopters, none for awf | Works and keeps awf's own type-check byte-exact, but diverges awf's rendered bytes from adopters', breaking dogfood fidelity. |
| `@ts-nocheck` with no harness strip | Delivers zero noise, but concedes ADR-0123 Decision #7's static-type-check pillar: the dogfooded extension would no longer be type-checked at all. |
